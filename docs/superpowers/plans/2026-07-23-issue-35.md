# Restyle Plan in the silent-menu language (issue #35) — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Restyle the Plan screen (`/plan`) into the #28 silent-menu language — cream kick bands, chocolate ink, caramel micro-labels, red-brown focus — matching `design/plan-silent-menu-combined.html` (round-4 pick: V2 + the owner's section reorder), swap Adherence above Profile, and leave every data-viz colour exactly as it is.

**Architecture:** Mostly a restyle. Three regions of `app/globals.css` are rewritten block-by-block from the cool-grey `--ink`/`--muted`/`--faint`/`--border` palette to the shared `--sm-*` silent-menu palette. `app/plan/page.tsx` changes structurally (section order, kick-band markup, enter stagger). The **one non-cosmetic change** is a new `proteinG` field on `PlanMealView` in `lib/plan.ts`, needed for the caramel protein pill the Done-when asks for on meal headers — it does not exist in the view model today.

**Tech Stack:** Next.js (App Router, the repo's forked version), React server + client components, vanilla CSS with `@theme` tokens in `app/globals.css`, vitest against live Neon.

## Open decisions (owner: confirm at the plan gate)

Decisions 1 and 2 are the ones most likely to be wrong.

**1. The caramel protein pill needs a data change that the ticket does not mention.**
The Done-when asks meal headers to "gain chocolate kcal + caramel protein pills". `PlanMealView` (`lib/plan.ts:22-29`) carries `kcal` but **no protein**, and `PlanItemView` has no per-item protein either — so there is nothing to render a protein pill from today. There is also a live-edit wrinkle: while a meal is being edited, the kcal figure is recomputed client-side from `it.unitKcal` (`meal-plan-editor.tsx:36`), and no equivalent protein number exists in the editor's state.
**Recommendation, and what this plan assumes:** add `proteinG` to `PlanMealView` only, computed server-side by reusing the existing `sumRawMacros` helper on the meal's own rows (one line, same round-once rule), **and hide the protein pill while that meal is being edited** — the kcal pill keeps its live recompute, the protein pill returns on save. This costs 1 field + 1 test and touches no client state.
*Alternative if the owner wants a live protein pill during edit:* thread `proteinG` through `PlanItemView` → `page.tsx`'s `overridesByMeal` mapping → the editor's item state as `unitProteinG`, and mirror the `pendingMealKcal` arithmetic. That is 3 more files and real edit-path logic in a ticket labelled "restyle"; I'd file it separately.
*Alternative if the owner wants zero data change:* ship the chocolate kcal pill only and file the protein pill as its own ticket. The combined spec never shows meal rows (they are collapsed there), so nothing in the accepted mockup would look wrong.

**2. No chevrons on the Meal plan / Memory bands.**
The combined spec draws `▾` on those two bands — but that chevron is the affordance for the collapse behaviour in **#36**, which is not built. **Recommendation, and what this plan assumes:** ship the bands without chevrons. A chevron that does not collapse anything is a lie to the user, and #36 will add it together with the behaviour that justifies it. If the owner would rather see the chevron now as a static glyph, it is a one-span addition to each band.

**3. The Adherence section loses its white `.plan-card` wrapper.**
V2 puts the adherence strip directly on the cream canvas under its band — no white card, no border. **Recommendation, and what this plan assumes:** drop `<div className="plan-card">` from `page.tsx` and delete the now-orphaned `.plan-card` rule (it has exactly one use site, `page.tsx:72`). If the owner wants the card kept, it needs warming to `--sm-border` instead.

**4. `.btn-dark` is warmed only inside `.plan`, not globally.**
`.btn-dark` (`globals.css:827`) is shared with Groceries and Chat and is still cool `var(--ink)`. Warming it globally would change two already-accepted screens. **Recommendation, and what this plan assumes:** scope the warm variant as `.plan .btn-dark`. The leftover cool button on Groceries' nutrition-lookup row is then a genuine loose end — worth a follow-up issue alongside the existing #37 cleanup ticket, not a change here.

**5. One new token, `--sm-ink-soft: #8a7a66`.**
V2 uses this warm mid-brown twice (the adherence hero sub-label and the food amount column). It is not in the token set, and the nearest existing token `--sm-muted` (#b3a390) is too light to preserve the hierarchy between an amount and its meta line. **Recommendation, and what this plan assumes:** add `--sm-ink-soft` to the silent-menu block, mirroring how the existing `--ink-soft` sits between `--ink` and `--muted`.
