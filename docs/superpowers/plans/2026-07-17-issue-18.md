# Issue #18: ⇄ Meal Edit Chooser Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Tapping Edit on a plan-screen meal that has a today-override (⇄) first asks "Edit today's ⇄ version" or "Edit the everyday meal", then seeds the editor from the chosen source, so a "Save for today" can no longer silently discard the existing override.

**Architecture:** Pure client + page wiring; zero API changes. `app/plan/page.tsx` already fetches `getOverridesForDate(today)` and passes only the meal ids down; it will now pass the override items themselves (shaped like the editor's `EditItem`). The editor gains a chooser state for ⇄ meals; the chosen branch seeds the rows and fixes the save scope, replacing the Apply switch for that edit. `lib/overrides.ts` needs one additive field (`servingGrams`) so override lines carry everything an `EditItem` needs.

**Tech Stack:** Next 16 App Router (RSC page → client component), Drizzle + Neon (live DB in tests), vitest.

## Owner decisions already made (2026-07-17, via interactive demo)

The owner picked these by playing with `design-directions/issue-18-meal-editor.html`
(untracked, git-excluded; regenerate nothing, decisions are recorded here):

- **Direction 1D**: Edit on a ⇄ meal opens a chooser (today's version / everyday meal), not the editor directly.
- **Question 2 (Apply switch) is moot by construction**: the chooser fixes where the save goes, so the Apply segment is hidden for chooser-initiated edits. Non-⇄ meals keep the switch and behave exactly as before.
- **Done when** (owner-approved wording, adapted to 1D): Editing a ⇄-adjusted meal offers today's override as a seed; "Save for today" updates the override without reverting to template contents; meals without an override behave exactly as before; `npx tsc --noEmit` and the vitest suite stay green.
